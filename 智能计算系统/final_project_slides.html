03 RAG 引擎 — 从问题到上下文

这部分的重点是：不用向量库，也能把课程材料整理成可检索、可排序、可塞进 Prompt 的上下文。rag_engine 负责把原始资料变成 chunk，再根据问题挑出最相关的片段。

1. 构建 Chunk

读取 txt/jsonl，记录 source、page、section、text，形成统一文档片段。

2. 解析问题

抽取中文 token、短语、领域关键词，并判断可能属于哪个课程主题。

3. 融合打分

BM25 分数作为主体，再叠加关键词、主题画像和来源权重。

4. 组装上下文

去重、限制每源数量、控制总字数，只把 Top 片段交给 Prompt。

问题→词汇检索→主题加权→片段去重→带来源的 RAG 上下文

04 RAG 引擎 — 知识库数据样例

知识库分成两层：txt 课程笔记负责高密度知识点，jsonl 教材片段负责覆盖面和页码溯源。检索时两类数据会被统一成 chunk 参与排序。

格式内容分块策略
.txt课程笔记 (4 个主题文件)按【】/=== 标题分割 section → 段落分块 ≤800 字符
.jsonl教材全文摘录 (493 条)逐行解析 JSON，保留页码、章节元信息

.txt 课程笔记片段高权重

【风格迁移】
内容图像用于约束生成图像的主体结构，风格图像用于提供纹理、色彩和笔触特征。
内容损失一般取 VGG19 高层特征的均方误差，强调语义结构一致。
风格损失使用 Gram 矩阵表示特征通道之间的相关性，适合描述纹理统计。

【RNN / LSTM / GRU】
RNN 通过隐藏状态传递历史信息，但长序列训练中容易出现梯度消失。
LSTM 引入输入门、遗忘门和输出门，GRU 用更新门和重置门简化结构。

07 RAG 上下文注入 — 证据如何进入 Prompt

证据格式统一

每条资料保留 source、page、section 和正文内容，注入时用【资料】前缀区分普通指令与课程材料。

资料优先级约束

显式要求回答优先依据检索资料，避免模型直接凭通用知识扩写，降低课程范围外发挥。

问题保持原样

用户问题不和资料硬拼接，RAG 上下文单独注入，便于区分“要回答什么”和“依据什么回答”。

这一页的重点是 RAG 的“最后一公里”：检索结果不是简单贴到问题后面，而是作为带来源的证据块进入上下文，让回答更贴近课程资料。

08 RAG 输出 — 让检索结果变成稳定答案

证据驱动回答

资料优先 限定课程范围 保留来源线索 总结收束

回答不是“自由发挥”，而是围绕 Top 片段组织语言：先覆盖检索到的核心概念，再用简短总结收尾。

RAG 结果清理

  1. 去掉重复资料标题
  2. 合并相似解释句
  3. 保留公式与关键术语
  4. 控制答案篇幅
  5. 补齐总结句
  6. 空检索时给出兜底说明

覆盖面广 · 带页码元信息 · 可溯源

设计哲学：笔记层信息密度高、噪声低 → 赋予更高权重；教材层覆盖面广但粒度粗 → 兜底补充。主题画像的来源映射利用了这一分层结构（如"风格迁移"优先匹配 style_transfer.txt）。

10 RAG 设计亮点 (1/2)

✨ 分层知识库

txt 课程笔记作为高置信知识层，jsonl 教材全文作为覆盖层；检索时统一成 chunk，但保留不同来源权重。

✨ 轻量词汇级 RAG

不依赖嵌入模型和向量数据库，用 BM25 + 课程关键词完成限定领域检索，部署成本低、逻辑透明。

✨ 多信号融合排序

BM25、领域关键词、中文短语、主题画像、来源权重一起打分，避免只靠单一词频导致结果偏移。

✨ 主题画像增强

把风格迁移、RNN、Transformer、处理器架构等课程主题做成画像，问题命中主题后优先召回对应资料。
