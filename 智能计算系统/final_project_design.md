# 智能计算系统大作业——设计思路与亮点

## 一、项目概述

本项目是一个**完全本地化运行的 RAG（检索增强生成）课程问答系统**，针对"智能计算系统"课程设计。系统以 Qwen2.5-1.5B-Instruct 作为基座语言模型，结合自建课程知识库，在不依赖任何网络调用、向量数据库或嵌入模型的前提下，实现高质量的中文课程问题解答。

核心设计目标：

- **全离线运行**：模型、分词器、知识库均为本地文件，无网络依赖
- **检索精准**：在无嵌入模型的约束下，通过多重信号融合的词汇级检索实现高召回率
- **输出可控**：通过 prompt 工程和后处理管线保证答案格式规范、内容可靠
- **资源友好**：支持 int8 量化加载，单次推理后主动释放 GPU 显存

---

## 二、整体架构

```
CLI参数(问题 + 可选知识库路径)
        │
        ▼
    main.py  ── 编排器
        │
        ├── model_loader.py  ── 模型与分词器加载（全局缓存）
        ├── rag_engine.py    ── 知识库检索（BM25 + 多信号增强）
        ├── prompt_builder.py ── Chat 模板构建（系统提示 + RAG上下文注入）
        └── inference.py     ── 贪心解码 + 后处理去重/总结补全
```

系统采用**模块化管道架构**，各模块职责单一、接口清晰。`main.py` 作为编排器按顺序调度各模块，模块之间通过明确的输入输出接口解耦，便于独立测试和替换。

## 三、核心模块设计

### 3.1 模型加载模块（model_loader.py）

**两级加载策略**：

1. **优先 int8 量化**：通过 bitsandbytes 的 `BitsAndBytesConfig(load_in_8bit=True)` 加载，可节省 40-45% 显存
2. **回退 fp16**：若 bitsandbytes 不可用（如 Windows 环境），自动回退为 `torch.float16` + SDPA 高效注意力

**全局模块级缓存**：`_model` 和 `_tokenizer` 作为模块级全局变量，首次调用加载模型后缓存，评测器多次调用 `main()` 时无需重复加载，避免重复 I/O 和 GPU 显存分配。

**关键细节**：

- Tokenizer 设置 `padding_side="left"`，适配 decoder-only 模型的 batch=1 生成场景
- 使用 `device_map="auto"` 自动将模型分布到可用 GPU
- `low_cpu_mem_usage=True` 减少加载时的 CPU 内存峰值
- 统一使用 SDPA（`attn_implementation="sdpa"`）注意力实现，利用 PyTorch 2.0+ 内置优化

### 3.2 RAG 检索引擎（rag_engine.py）

这是系统最核心的模块，实现了一套**纯词汇级的 BM25 风格检索器**，完全不依赖嵌入模型或向量数据库。

#### 3.2.1 知识库构建

支持两种文件格式的混合加载：

| 格式       | 内容           | 分块策略                                      |
| -------- | ------------ | ----------------------------------------- |
| `.txt`   | 课程笔记（4个主题文件） | 按 `【】`/`===` 标题分割 section，再按段落分块，每块≤800字符 |
| `.jsonl` | 教材全文摘录（493条） | 逐行解析 JSON，保留页码、章节等元信息                     |

所有文本统一经过 **NFKC Unicode 归一化**处理，并包含一套自定义的**康熙部首替换表**（将 16 个部首字符映射为标准汉字，如 ⺠→民、⻄→西），以及希腊字母展开（α→alpha, β→beta 等），确保中文文本比较的一致性。

#### 3.2.2 多重信号融合的检索评分

检索评分由以下五个信号源加权合成：

**信号一：Token 级 BM25 评分**

- 对中文采用 unigram + bigram 分词，对拉丁字符采用正则分词
- 使用 BM25 公式（k1=1.4, b=0.75）：`score = IDF(term) * tf * (k1+1) / (tf + k1*(1-b + b*dl/avgdl))`
- 高频查询词获得额外加权（+0.15 × min(qf,3) × IDF）

**信号二：领域关键词匹配加成（+4.0 + len×0.35/词）**

- 预定义约 150 个 `DOMAIN_KEYWORDS`，覆盖 RNN/LSTM/GRU、CNN/风格迁移、Transformer、深度学习处理器、编程框架、调试调优、大模型等全部课程主题
- 关键词按章节分组注释，便于维护和扩展

**信号三：中文短语匹配加成（+3.0 + len×0.25/短语）**

- 从问题中提取 4-8 字中文连续子串，在文档的去空格版本中做包含匹配
- 捕捉 BM25 单 token 无法覆盖的多字组合语义

**信号四：主题画像加成**

- 定义了 6 个主题画像（style_transfer、rnn、transformer、processor、debug_tuning、framework），每个画像包含问题特征词和文档特征词
- 当问题命中某主题时，来源文件匹配的文档获得 +3.0 基础加成，文档内主题关键词命中额外累加（最多 +4.0）

**信号五：来源权重系数**

- `style_transfer.txt` 和 `rnn_experiment.txt`（实验相关，精准度高）：权重 1.8
- `dl_fundamentals.txt`（基础通用知识）：权重 1.3
- `advanced_topics.txt`（进阶教材）：权重 1.15
- 教材 JSONL（通用背景知识）：权重 1.0

#### 3.2.3 结果构建的去重与多样性策略

- **完全去重**：按 `(source, page_start, text[:80])` 三元组去重
- **来源均衡**：无页码的 txt 文件每个来源最多入选 2 条，JSONL 每个来源最多 1 条
- **总长控制**：所有片段总字符数不超过 `MAX_RAG_TOTAL_CHARS`（2000），单条不超过 `MAX_RAG_CHARS`（800）
- 默认返回 top-3 条最相关片段

### 3.3 Prompt 构建模块（prompt_builder.py）

构建三层 chat 消息结构：

1. **系统提示**：定义助教角色、回答规则（200-350字、最多5要点、每点一句话）、课程范围、格式要求（以"总结："结尾）
2. **RAG 上下文注入**：将检索到的知识片段以 `【资料】` 前缀注入第二个 system 消息，标注来源、页码和章节，并明确指示"资料优先级高于通用知识"
3. **用户问题**：原始问题文本

这种设计让模型明确区分"系统行为约束"和"参考知识素材"，比将 RAG 内容直接拼入用户消息更符合 Qwen2.5 的 chat template 设计意图。

### 3.4 推理与后处理模块（inference.py）

#### 3.4.1 生成策略

采用**贪心解码**（`do_sample=False`），不使用温度采样。这保证输出确定性、速度最快，适合课程问答场景（答案应稳定、可复现）。同时配置：

- `repetition_penalty=1.15`：抑制重复生成
- `no_repeat_ngram_size=6`：阻止 6-gram 级别重复
- 推理使用 `torch.inference_mode()` 而非 `no_grad()`，获得更彻底的计算图禁用

#### 3.4.2 后处理管线

后处理是保证输出质量的关键环节，包含以下步骤：

1. **空白字符归一化**
2. **重复标题去重**：通过正则识别"一、"/"1."等编号标题模式，将语义相同的标题合并
3. **句子级去重**：
   - 先做 O(1) 精确集合查重
   - 再做子串包含检查（长度≥16 时）
   - 最后做模糊匹配（`SequenceMatcher.ratio() >= 0.94`，仅当长度比≥0.7 时触发，避免性能浪费）
   - 已见句子列表上限 60 条，超过后只保留最近 40 条，防止无限增长
4. **公式检测跳过**：含数学符号或 LaTeX 的句子跳过去重逻辑，保护公式完整性
5. **总结强制补全**：若模型未生成"总结："行，自动提取最后一句实质性内容作为兜底总结
6. **首条总结截断**：若"总结："出现在中间，截断其后的冗余内容
7. **空输出兜底**：若最终答案为空，返回通用兜底回答，确保评测器不会收到空输出

## 四、知识库设计

知识库是 RAG 系统的信息基础，本项目精心设计了**分层知识库**结构：

| 层级    | 文件                                                                                    | 用途                                 |
| ----- | ------------------------------------------------------------------------------------- | ---------------------------------- |
| 课程笔记层 | `style_transfer.txt`、`rnn_experiment.txt`、`dl_fundamentals.txt`、`advanced_topics.txt` | 由课程知识点提炼而成，结构清晰、公式精确，是检索的**高权重信源** |
| 教材全文层 | `book_chunks.jsonl`（493条）                                                             | 教材原文摘录，按章节标注页码，作为**通用背景补充**        |

设计考量：

- 笔记层与教材层分层管理：笔记信息密度高、噪声低，赋予更高权重；教材覆盖面广但粒度粗，作为兜底补充
- JSONL 格式支持页码和章节元信息，使 RAG 上下文可溯源，增强答案可信度
- 主题画像系统的来源映射（如"风格迁移"问题优先匹配 `style_transfer.txt`）利用了这一分层结构

## 五、设计亮点总结

### 亮点一：零依赖的轻量级 RAG 检索

在没有任何嵌入模型（如 text2vec、BGE）和向量数据库（如 FAISS、Chroma）的情况下，纯靠词汇级 BM25 检索 + 领域知识增强，实现了有效的课程知识检索。整个 RAG 引擎仅依赖 Python 标准库（`re`、`json`、`math`、`collections.Counter`），不需要任何额外的 pip 依赖。

### 亮点二：多重信号融合的检索评分

单一 BM25 评分在中文短问题场景下召回率有限。本系统的五重信号融合策略（BM25 + 领域关键词 + 中文短语 + 主题画像 + 来源权重）充分利用了课程知识库的结构化特征（已知主题分布、已知来源可靠性），在限定领域内显著提升了检索精度。

### 亮点三：精心设计的后处理管线

大模型生成存在天然的不确定性——可能产生重复句子、遗漏格式要求、甚至输出空内容。本系统的后处理管线通过 O(1) 精确去重 → 子串检测 → 模糊匹配的**三级去重漏斗**，以及总结强制补全机制，将不可控的模型输出转化为格式规范、内容紧凑的最终答案。

### 亮点四：对抗式鲁棒性设计

系统中多处体现了对边缘情况的防御性处理：

- 模型加载的两级回退（int8 → fp16）
- RAG 检索的空知识库保护（返回空列表不会导致崩溃）
- 空答案兜底（绝不输出空字符串给评测器）
- Unicode 部首归一化（处理特殊编码的中文字符）
- 已见句子列表上限（防止长时间运行内存泄漏）

### 亮点五：显存精细管理

- int8 量化优先加载路径
- 生成后立即 `del inputs`、`del outputs`
- 主动调用 `gc.collect()` 和 `torch.cuda.empty_cache()`
- 全局模型缓存避免重复加载
- 这些措施使得 1.5B 模型在消费级 GPU（甚至 CPU）上也能稳定运行

### 亮点六：模块化可扩展架构

每个模块通过明确的输入输出接口解耦。若需要升级为向量检索，只需替换 `rag_engine.py`，其余模块无需修改。若需切换为更大的模型，只需修改 `model_loader.py` 中的模型路径。Config 集中的参数管理使得调参和实验变得高效。

### 亮点七：课程知识的结构化工程

知识库的构建不仅仅是将文本"丢进去"，而是：

- 笔记文件按主题拆分、按标题分 section、按段落分 chunk
- 关键词库按课程章节分组注释
- 主题画像建立"问题词 → 文件来源 → 文档词"的三级映射
- JSONL 保留页码元信息支持答案溯源

这些工程化处理使得一个纯词汇级的检索器也能达到接近语义检索的效果。
